# T2.5 — Signed Updates: Threat Model & Design Memo

> **Session:** T2.5 (Signed Updates, Ed25519) · **Status:** BUILD · **Date:** 2026-06-02
> **Spec:** `specs/T2-production-systems-design.md` §7 (Subsystem F) + §8 (key management) — **APPROVED 2026-05-31**
> **Branches:** code → `audit/T2.5-signed-updates` (Development repo); docs → `audit/T2.5-findings` (root repo)
> **Depends on (merged/stacked):** T1.6 (`release.ps1` `VERSION`/`-CheckSync` refactor — in `main`), T2.3 (`lib/ed25519.ahk` shared verifier — stacked below this branch).

This memo is the Phase-0 deliverable: it pins the attack surface, what the signature does and does **not** defend, the exact wire schema + canonicalization recipe (so a future session reproduces it byte-for-byte), the verify mechanism and its OS floor, and key storage/rotation.

---

## 1. What is being defended

QuickSay auto-updates by fetching `https://quicksay.app/version.json` in `CheckForUpdates()` (`QuickSay.ahk` ~3663) and, if the remote `version` is newer, offering the user a download (`Run(download_url)` after a bare `^https://` check). **Today there is no authentication of the manifest at all** — the `^https://` prefix check is the only "security." Any party who can:

- write to the R2 bucket / origin that serves `version.json` (ACL slip, credential leak, insider), or
- MITM the TLS connection (corporate proxy, hostile network, a future TLS break),

can serve an attacker-chosen `version` + `download_url` and steer the user to a malicious "update." This is a classic software-update supply-chain exposure.

**Goal:** the app installs/offers an update only when it can *cryptographically prove* the manifest came from us. Achieved by signing `version.json` with our Ed25519 key in `release.ps1`, and verifying that signature in `CheckForUpdates()` before trusting **any** field. Unsigned / tampered / wrong-key / unknown-`keyId` manifests are rejected and the app **fails closed** (no update offered).

## 2. Threat enumeration (`ultrathink`)

### 2.1 Attacks the signature STOPS

| # | Attack | Why it's stopped |
|---|--------|------------------|
| A1 | R2/origin write-compromise serves a malicious `version.json` | Attacker lacks the private key → can't produce a valid signature over the tampered payload → verify fails → fail closed. |
| A2 | MITM rewrites the manifest in flight (TLS strip/break, hostile proxy) | Same — any byte change to a signed field breaks the Ed25519 verification. |
| A3 | Typo/ACL mistake exposes the bucket to public write | Same — signature is the trust anchor, not the bucket ACL. |
| A4 | Attacker points the app at a fake `download_url` (steer to malware) | `download_url` is **inside** the signed payload → tampering it invalidates the signature → no download offered. |
| A5 | Attacker keeps a valid (older) manifest but swaps the *installer* at `download_url` | Partially — the manifest commits to `installer_sha256` (signed). The hash is the cryptographic commitment; **enforcement is pending in v2.0** (see §2.3 / F6). |
| A6 | Downgrade to an older *signed* manifest (rollback) | Out of the signature's job, but neutralised in practice: `CompareVersions` only offers an update when remote `version > local`, so a replayed older signed manifest yields "you're up to date," never a forced downgrade. (No anti-rollback counter is needed because updates are user-confirmed and never auto-applied.) |
| A7 | `alg`/algorithm-confusion or `alg:none`-style downgrade | N/A by construction — `version.json` carries **no `alg` field**; the verifier hardcodes Ed25519 and never reads an algorithm hint from the manifest (spec §3.4 / §7.3). |

### 2.2 Attacks the signature does NOT stop (accepted / out of scope)

| # | Attack | Disposition |
|---|--------|-------------|
| N1 | **Compromised signing key** | If `ED25519_PRIVATE_KEY` leaks, signatures are worthless. Mitigated by key storage discipline (§4): CF secret store + offline backup, **never in git**, and the local generator copy deleted at M.1 (gate I-a). Recovery = key rotation (§5). |
| N2 | **Local malware patches the app on disk** | OS-integrity problem, out of scope. The compiled-in public key can itself be patched by code that already has write access to the binary. Defense is Azure code-signing + OS controls, not this mechanism. |
| N3 | **Installer bytes swapped behind a valid manifest, pre-enforcement** | v2.0 *commits to* `installer_sha256` but does **not** download-and-hash the installer (the app opens `download_url` in the browser). Binary integrity in v2.0 therefore rests on **Azure code-signing + TLS**, not yet on the manifest. F6 / §2.3. |
| N4 | **Sentry/Worker/other backend compromise** | Unrelated to update signing. |
| N5 | **Stolen public key** | The public key is *meant* to be public (it's the trust anchor baked into every binary). No confidentiality requirement. |

### 2.3 The installer-integrity gap (F6 — "mitigated-pending")

Signing only the manifest while leaving the installer unauthenticated would let an attacker keep a valid manifest and swap the binary at `download_url` (A5). The manifest therefore **commits to `installer_sha256`** (the SHA-256 of the *final, code-signed* installer), and that field is inside the signed payload.

**However**, in v2.0 `CheckForUpdates()` only opens `download_url` in the user's browser; it does not download the installer itself, so it cannot compare bytes against `installer_sha256`. Consequently the hash is **committed-to but not enforced** by the app yet. The bytes the user actually runs are guarded by **Azure code-signing (SmartScreen) + TLS**.

**Contract for the future (M.1 candidate):** if/when the app downloads the installer itself, it MUST verify `SHA-256(downloaded bytes) == signed installer_sha256` **before executing, fail-closed**. Until then, `CheckForUpdates()` surfaces/logs the expected hash. (Flagged out-of-scope for M.1.)

**Note on `download_url`:** the shipped manifest's `download_url` currently points at the `https://quicksay.app/download` landing page, while `installer_sha256` is the hash of the concrete versioned installer artifact (`QuickSay_Beta_v<maj.min>_Setup.exe`) uploaded to R2 + attached to the GitHub release. The hash is the authoritative commitment to "the installer for this version." When M.1 wires in-app download, it must fetch the concrete artifact URL and hash-check it. This URL-vs-artifact distinction is recorded so M.1 closes it deliberately.

## 3. Wire format & canonicalization (reproduce byte-for-byte)

### 3.1 Signed `version.json` schema (additive — old apps ignore the new fields)

```json
{
  "version": "2.0.0",
  "download_url": "https://quicksay.app/download",
  "changelog": ["First line", "Second line é✅ / ok"],
  "installer_sha256": "<hex sha256 of the final signed installer>",
  "released_at": "2026-06-02T00:00:00Z",
  "keyId": "qs-2026",
  "signature": "<base64url, no padding, of the 64-byte Ed25519 signature>"
}
```

Old (pre-T2.5) apps read only `version`/`download_url`/`changelog` and ignore the rest — they keep the pre-T2.5 (unverified) behavior, which is acceptable and additive. The `release_date` field used by the pre-T2.5 script is replaced by `released_at` (ISO-8601, signed); nothing consumed `release_date` (verified: no Website or app reader).

### 3.2 Canonical signed payload (the exact bytes that get signed/verified)

The **signed payload** is the manifest **excluding `signature`**, serialized as a single JSON object containing **exactly** these six keys:

```
changelog, download_url, installer_sha256, keyId, released_at, version
```

Rules (spec §7.2, finding F5/D20 — signer `JSON.stringify` ≡ verifier must agree byte-for-byte):

1. **Keys sorted** lexicographically by ASCII code point: `changelog < download_url < installer_sha256 < keyId < released_at < version`.
2. **Compact separators**: `,` between members and `:` between key/value, **no insignificant whitespace**.
3. **Minimal RFC 8259 string escaping only**: escape `"` → `\"`, `\` → `\\`, and control chars U+0000–U+001F (short forms `\b \t \n \f \r`, else `\u00XX`). **Do NOT escape `/`.** **Do NOT `\u`-escape non-ASCII** — emit raw UTF-8 bytes (an emoji/accented changelog entry is its literal UTF-8 bytes). This matches Node `JSON.stringify`.
4. `changelog` array element order is **preserved** (not sorted).
5. The result is encoded as **UTF-8 bytes**; Ed25519 signs those bytes. `signature = base64url(no padding)` of the 64-byte result.

Concretely, for the example above the canonical string is:

```
{"changelog":["First line","Second line é✅ / ok"],"download_url":"https://quicksay.app/download","installer_sha256":"<hex>","keyId":"qs-2026","released_at":"2026-06-02T00:00:00Z","version":"2.0.0"}
```

(the `é`, `✅`, and `/` appear literally — no escaping).

**Signer** (`release.ps1` → Node) and **verifier** (`lib/update-verify.ahk`) both build this exact string. The verifier parses the fetched JSON, then **reconstructs** the canonical string from the parsed field values using the same recipe and Ed25519-verifies. Test #9 (byte-identity) MUST include a non-ASCII changelog entry (emoji + accented char + `/`) to prove the two serializers agree; test #8 is the RFC 8032 known-answer vector proving the Ed25519 implementation itself.

## 4. Verify mechanism & key management

### 4.1 Mechanism (Phase-1 "riskiest call" — resolved)

The Ed25519 verification primitive is **the shared pure-AHK verifier `lib/ed25519.ahk`** (`VerifyEd25519(msg, sig, pubKey)`), created by T2.3 and imported here per spec §8.1 ("whichever session lands first creates it; the other imports it"). It is a from-RFC-8032 edwards25519 implementation (BCrypt only for SHA-512/256), so it has **no Windows-build floor** beyond what QuickSay already requires (Win10+ for BCrypt SHA-2, universally present). This sidesteps the CNG-Ed25519 availability question entirely (CNG Ed25519 is not reliably present on the Win10 floor).

**Already proven** (T2.3 `tests/license/ed25519-tests.ahk`, passing): SHA-512 KAT, **RFC 8032 §7.1 TEST 2 KAT**, Node-interop with a non-ASCII message, tamper/wrong-key rejection, malformed-input fail-closed, and the **I-c build-time trust-anchor assert** (`SHA-256(qs-2026 raw pubkey) == 761d22df…fde09b`). T2.5 re-uses this verifier unchanged and adds manifest-level tests on top.

### 4.2 Keys (spec §8.1 — one shared key for JWTs and updates)

```
keyId            : qs-2026
publicKeySha256  : 761d22dfcc2302fe1364bc36ec76b7aa488c666adc828f0247e866bf80fde09b
raw-32 base64url : UmeruJlXQ1tyEX5fPzixUMjQD__Lm0NqIPSWReHRsw8   (baked into the app as a trust anchor)
SPKI PEM         : MCowBQYDK2VwAyEAUmeruJlXQ1tyEX5fPzixUMjQD//Lm0NqIPSWReHRsw8=
```

- **Public key** → committed and compiled into the app as `TRUSTED_UPDATE_KEYS["qs-2026"]` (`lib/update-verify.ahk`). Public by design.
- **Private key** (PKCS#8 PEM) → CF secret `ED25519_PRIVATE_KEY` **+** offline backup (1Password/equivalent). **Never in git.** At sign time, `release.ps1` reads it from a secret source — env var `QUICKSAY_ED25519_PRIVATE_KEY` (PEM contents) or `QUICKSAY_ED25519_PRIVATE_KEY_PATH` (a file path outside the repo; default `~/.quicksay-keys/qs-2026-ed25519-private.pem`). The signer **fails loudly** if the key is absent — no unsigned release ships.
- **I-a (M.1 gate):** the local generator copy at `%USERPROFILE%\.quicksay-keys\` must be deleted before rc1, after confirming it lives in the CF secret store + offline backup. The most likely real-world compromise path for the key that signs both licenses and updates is a lingering local copy.
- `.gitignore` excludes `*.pem`, `*-private.pem`, and `.quicksay-keys/` so a stray key can never be staged.

### 4.3 Key rotation (spec §7.4 / §11 Q3 — rotate only on compromise)

One shared key (`qs-2026`) for staging + production; no scheduled rotation. The `keyId → publicKey` map is the non-breaking rotation mechanism:

1. Generate a new keypair under a new `keyId` (e.g. `qs-2027`).
2. Ship an app release whose `TRUSTED_UPDATE_KEYS` map contains **both** old and new public keys.
3. Once that release is widely adopted, switch the Worker (`ED25519_PRIVATE_KEY`) and `release.ps1` to sign with the **new** key.
4. Retire the old `keyId` after a transition window of **≥1 release cycle (~90 days)** so nearly all installs trust the new key.

Because the app trusts a *map* of keys, adding a key is non-breaking and removing one only affects installs that never upgraded.

## 5. Fail-closed behavior (normative)

`CheckForUpdates()` treats the manifest as **untrusted → no update available** on any of:

- `signature` field missing / empty / not valid base64url / not 64 bytes after decode;
- `keyId` missing or not present in `TRUSTED_UPDATE_KEYS`;
- any of the six signed fields missing or malformed (can't rebuild the canonical payload);
- Ed25519 verification returns false (tampered any signed field, wrong key);
- JSON parse failure.

On rejection: do **not** call `Run(download_url)`, do **not** offer the download, **log the specific reason** to the debug log when `debugLogging` is on (e.g. `update rejected: signature invalid`, `update rejected: unknown keyId qs-old`). The **user-facing** message stays generic — *"Could not verify the update. Please download from quicksay.app."* — on a manual check only, so an attacker probing the check learns nothing. Never fail open.

## 6. Out-of-scope notes (flag for later)

- **F6 in-app installer download + hash-check** → M.1 candidate (§2.3). v2.0 commits-to but does not enforce `installer_sha256`.
- `download_url` points at the `/download` landing page, not the concrete artifact whose hash is signed — M.1 must fetch the concrete artifact when it adds in-app download (§2.3 note).
- `CheckForUpdates()` now reads the single `global localVersion := "1.9.0"` SSOT (rewritten by `release.ps1` + verified by `-CheckSync`) instead of a function-local literal, so the update check and the crash reporter's Sentry `release` tag can't drift.
