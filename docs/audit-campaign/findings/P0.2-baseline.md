## Re-verification of 6 Surface Inventory Items

### Item 1 — STT/LLM model hardcoding in UI

**Verdict: PARTIAL**

- `QuickSay.ahk:97` — `sttModel` loaded from config with fallback `"whisper-large-v3-turbo"`.  
- `QuickSay.ahk:98` — `llmModel` loaded from config with fallback `"llama-3.1-8b-instant"`.  
- Both are configurable via `config.json` and respected at runtime.
- `gui/settings.html` — no model selector UI present; the settings window exposes no control to change these values.
- **Finding:** The config system supports model configuration, but there is no UI surface to change models. A user cannot change the STT or LLM model through the settings window.

**Owner:** T1.3 (settings UI completeness audit)

---

### Item 2 — Context-aware mode switching may not be wired up

**Verdict: REFUTED**

- `QuickSay.ahk:1379-1435` — `GetContextModeId()` is fully implemented. It reads `contextAwareModes` from config and maps foreground process names (e.g., `code.exe`, `OUTLOOK.EXE`, `slack.exe`) to mode IDs.
- `QuickSay.ahk:2696` — `GetContextModeId()` is called inside `StartRecording()` before every transcription run.
- The feature is disabled by default (`contextAwareModes` is an empty object `{}` in default config), which is why it appeared inactive.
- **Finding:** Fully implemented and wired up. Off by default, which is correct behavior.

**Owner:** None — no action needed.

---

### Item 3 — `keepLastRecordings` config field is loaded but has no consumer

**Verdict: CONFIRMED**

- `QuickSay.ahk:1655` — `keepLastRecordings` is parsed in `ParseConfig()` and stored in the global map.
- Searched `QuickSay.ahk`, `lib/settings-ui.ahk`, `lib/http.ahk` for any read of `keepLastRecordings` beyond the parse site: none found.
- No audio file rotation logic exists anywhere in the codebase.
- **Finding:** `keepLastRecordings` is an orphaned config key — loaded, exposed in settings UI, but never acted upon. Audio files accumulate without bound.

**Owner:** T1.5 (audio file rotation / data hygiene)

---

### Item 4 — File write safety (AtomicWriteFile + mutex coverage)

**Verdict: PARTIAL**

- `AtomicWriteFile()` is used for config.json, history.json, statistics.json, and dictionary.json — all critical data files. This is correct.
- `AcquireConfigLock()` (named mutex `QuickSay_ConfigLock`) is used in `HandleSaveConfig()` (`lib/settings-ui.ahk:949`) — correct.
- `lib/settings-ui.ahk:1549` — `SaveJSON()` (used by dictionary save) uses `AcquireConfigLock()` + `AtomicWriteFile()`.
- **Gap 1:** `SaveToHistory()` (`QuickSay.ahk:1880`) calls `AtomicWriteFile` but does NOT hold `AcquireConfigLock()`. Since only the tray process writes history, the mutex absence is acceptable — but the omission is inconsistent.
- **Gap 2:** `ShowWeeklySummary()` (`QuickSay.ahk:620`) reads statistics.json without a lock. Only reads — no write race possible — but noted for completeness.
- **Finding:** Atomic writes are correctly applied. Mutex coverage is nearly complete; `SaveToHistory()` is the only write path without a lock, and it is single-writer so the risk is low.

**Owner:** T1.6 (concurrency / data integrity audit)

---

### Item 5 — Version string mismatch across files

**Verdict: CONFIRMED**

- `QuickSay.ahk:44` — `localVersion := "1.9.0"` 
- `setup.iss` — `AppVersion=1.9.0`
- `signing/metadata.json` — `"Version": "1.9.0.0"` (four-part Windows version format — expected difference)
- `data/changelog.json` top entry — `"version": "1.9.0-beta"` (stale beta suffix)
- `config.json` — `"lastSeenVersion": "1.9.0-beta"` (stale beta suffix)
- **Finding:** The four-part signing metadata version is intentional and correct. The `1.9.0-beta` suffix in `changelog.json` and `config.json` is stale — it should have been cleared to `1.9.0` at the time of the stable release. `release.ps1` updates `localVersion` and `AppVersion` but does not strip the `-beta` suffix from changelog entries or `lastSeenVersion` in config.json.

**Owner:** T1.7 (release pipeline / version string hygiene)

---

### Item 6 — History clear race condition (settings sends PostMessage before tray reloads)

**Verdict: REFUTED — already fixed**

- `lib/settings-ui.ahk:1199` — `HandleClearHistory()` deletes `history.json`, then calls `PostMessage(0x5555, 1, 0)` to the tray process window `"QuickSay_TrayMode ahk_class AutoHotkey"`.
- `QuickSay.ahk:778` — `ReloadConfig()` (triggered by `0x5555`) resets `HistoryTextCache := []` and `HistoryCacheLoaded := false`, so the in-memory cache is flushed.
- This fix was applied in commit `2ac7632` ("Fix 3 bugs: LLM cleanup artifacts, history clear race condition, intermittent slowness").
- **Finding:** The race condition is resolved. No action needed.

**Owner:** None — fixed in commit 2ac7632.
