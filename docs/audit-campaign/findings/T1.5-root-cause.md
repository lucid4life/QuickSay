## 1. The three bugs, each traced to its origin

**Bug A — `historyRetention` is enforced, but the enforcement is corrupting.** Retention *is* applied (added in commit `2ac7632`) at `QuickSay.ahk:1854-1877`, but by **string-surgery on the raw concatenated JSON text**, not by parsing. The entry *count* regex `"id"\s*:` is actually safe — `EscapeJson` rewrites every user quote to `\"`, so dictated text cannot forge an unescaped `"id":`. But the *trim* regex `\}\s*,` is **not** safe: `EscapeJson` leaves `}` and `,` literal, so any transcript containing `},` (dictated code, JSON, "etc., and") manufactures a false entry boundary and truncates `history.json` mid-entry (T1.1-017). Worse, entries are **prepended** (newest-first, line 1850) while the trim consumes separators **from the front**, so when the file *does* exceed the cap the code discards the **newest** entries and keeps the oldest (T1.2-016). retention=0 ("unlimited") survives only by an off-by-one accident (`trimmed = trimCount` never holds because an N-entry array has only N−1 separators).

**Bug B — `keepLastRecordings` is truly orphaned.** It round-trips through `GetDefaultConfig` (`:1297`), `ParseConfig` (`:1655`) and the settings UI, but **no read site prunes `data/audio/`**. `StopAndProcess` copies the WAV (`:2862-2864`) when `save_recordings` is on and never rotates. 89 files on disk, cap of 10, zero enforcement.

**Bug C — clear-history resurrection.** `HistoryTextCache` (`:135`, a **tray-lifetime** global) is the **authoritative source for every history write**: `SaveToHistory` lazy-loads it once (`:1831-1838`), rebuilds from it (`:1841-1851`), and writes it back (`:1880-1882`). Clearing history (settings side) deletes the file and posts `0x5555` (`settings-ui.ahk:1217`), but the tray's **only** cache-invalidation path is the **deferred** `ReloadConfig` (`:215-217` → `:778-781`, fired at `-100 ms`). That races the **deferred** `SaveToHistory` (`:3084`, fired at `-1`). When a transcription's write fires after the file was deleted but before the cache is flushed — the likely order, and inevitable when the tray pump was blocked on the synchronous Whisper/LLM HTTP during the clear — `SaveToHistory` rebuilds the file from the still-populated cache and every "deleted" entry returns. `2ac7632` flushed the cache *on reload* but never stopped an already-scheduled write from rebuilding from it; that is exactly why T1.2-011 calls the fix incomplete.

## 2. ultrathink — the one shared root cause

The three are one bug wearing three hats: **`history.json` is never treated as the single source of truth at the moment of mutation.** The write path trusts an in-memory cache (Bug C) and mutates by splicing text instead of doing a real read-modify-write of parsed JSON (Bug A); and because no mutation point is treated as "the place where the on-disk state is reconciled," nobody added the audio-rotation step there either (Bug B).

Enforce **one invariant** — *every history mutation is `fresh read → JSON.Parse → modify → trim → AtomicWriteFile`, serialized by the config mutex* — and all three collapse:

- **Race (C):** there is no persistent cache to resurrect from. A write that fires after a clear re-reads the now-empty/absent file and cannot bring old entries back. Whoever writes last wins, and "last" is always correct (a clear after a write wipes; a write after a clear starts fresh).
- **Retention (A):** trimming becomes `keep the first N of a newest-first array` — no regex, no `},`-in-text corruption, oldest dropped, retention=0 = unlimited by construction. Migration is automatic: a bloated legacy array is parsed and sliced on the first append after upgrade.
- **Audio (B):** the mutation path is the obvious, single home for `PruneAudioDirectory` right after the WAV is saved.

**Why the orphans survived beta unnoticed:** the failure is silent and threshold-gated. `history.json` just grows until someone crosses the cap (nobody had: 24 entries here), and `saveAudioRecordings` defaults **off**, so the audio cap has no files to bite until a user turns saving on and then dictates 10+ times. The settings UI ("Keep last N recordings", a retention dropdown) *implies* the caps are real, so no one suspected they were inert.

## 3. Chosen fix shape (race = Phase-4c option **B**, hardened with a touch of **A**)

I chose **option B (re-read on every write)** as the structural fix because it is the only one that is *trivially correct across both process models* (in-process settings via `LaunchSettings`, and the external `QuickSay.exe --settings` process that the reproduction recipe actually exercises while the tray pump is blocked). I remove `HistoryTextCache` entirely — it was a marginal optimization on the **deferred**, off-hot-path write (the real "intermittent slowness" `2ac7632` fixed was the `CheckForUpdates` guard + retention default, not this cache), so dropping it costs nothing and eliminates the resurrection vector by construction.

Layered on top, a lightweight **generation guard** (option-A flavor) makes the "cleared history stays empty even against an in-flight write" semantics deterministic for test 9: a monotonic in-memory `HistoryGeneration` is bumped by a new synchronous `0x5556` ("history cleared") message; each recording captures the generation at start and the deferred write drops itself if the generation advanced (a clear superseded it). The structural re-read already guarantees old entries never come back in every ordering; the generation guard additionally suppresses the single in-flight entry the clear was meant to wipe. Both history mutation paths (`SaveToHistory` append, settings `HandleClearHistory` delete) take `AcquireConfigLock`, closing the cross-process read-modify-write window.

**Adjacent concurrency fixes folded in (assigned to this session):** `HandleImportConfig` writes `config.json` with no mutex (T1.2-008) → wrapped in `AcquireConfigLock`; the settings RMW handlers `LoadJSON(unlocked) → mutate → SaveJSON(locked full-replace)` lose updates against a concurrent tray write (T1.2-009) → routed through a new `UpdateConfigKeys` helper that holds the lock across the full load-modify-write. Also fixed in passing because it is the same code and the same root (entry-count vs line-count): `getHistoryCount` (T1.2-012) now counts parsed array entries.
