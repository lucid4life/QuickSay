### Top 3 findings in plain English

1. **Clearing history can resurrect "deleted" entries** (T1.2-011, P1). The tray keeps the entire history in an in-memory string cache that outlives every transcription. Clearing history deletes the file and asks the tray to drop that cache — but only via a 100 ms-deferred reload that races against the transcription's own deferred write. If a transcription's write fires in that window, it rebuilds the file from the stale cache and every cleared entry comes back.
2. **The "entries saved" count is more than 2× wrong** (T1.2-012, P2). The Processing tab counts physical *lines* in `history.json`, but the file is a pretty-printed JSON array with blank separator lines. A real 10-entry file is reported as "21 entries saved," while the History tab (which parses the array) correctly shows 10.
3. **Sub-label text fails WCAG AA contrast** (T1.2-019, P2, measured live). `--text-tertiary` (#72728c), used for hints, history meta ("5 min ago · claude"), and sub-labels, measures 3.54:1 on `--bg-surface` and 4.10:1 on `--bg-base` — both below the 4.5:1 AA floor for normal text.

## Category A — WebView2 bridge completeness

### T1.2-001 — All 33 actions have a live handler · **NO ISSUE** (evidence)
Cross-referenced every `postToAHK(` send site (`settings.html`) against the `switch action` dispatcher ([settings-ui.ahk:234-313](../../../Development/lib/settings-ui.ahk)). The set sent == the set handled == the 33 in the matrix. No dead actions, no dead handlers.

### T1.2-002 — Redundant `clearHistory` / `deleteHistoryFile` aliases · **P3** · owned by T1.2-followup
`deleteHistoryFile` (:282) and `clearHistory` (:255) dispatch to the same `HandleClearHistory`. The two HTML paths differ subtly: `clearHistory()` ([settings.html:1921](../../../Development/gui/settings.html)) relies on AHK's pushed `receiveHistoryClearResult`/`receiveHistoryData`; the History-tab path ([settings.html:1218-1230](../../../Development/gui/settings.html)) instead fires `deleteHistoryFile` then a **separate 500 ms `setTimeout(loadHistory)`**. Two code paths to keep in sync for one operation.
**Fix:** collapse to one action; have both buttons call the same JS function.

### T1.2-003 — Unknown/malformed payloads fail silently · **P2** · owned by T1.2-followup
The dispatcher has **no `default` case**; an unknown `action` is only `OutputDebug`-logged ([settings-ui.ahk:232](../../../Development/lib/settings-ui.ahk)) and silently dropped. Several handlers read `msg["data"]` with no `msg.Has("data")` guard — `saveDictionary` (:252), `saveModes` (:289), `setMode` (:291), `previewSound` (:293), `markChangelogSeen` (:297), `loadLegalDoc` (:285), `openUrl` (:265). A missing `data` key throws, caught by the outer `try/catch` ([settings-ui.ahk:314-316](../../../Development/lib/settings-ui.ahk)) and swallowed → the control silently no-ops with zero user feedback. (Other handlers — `testGroqAPI`, `saveConfig`, `loadMoreHistory`, `testHotkey` — *do* guard with `msg.Has`. Inconsistent.)
**Fix:** add a `default` branch that logs/telemetries the unknown action; guard all `msg["data"]` reads uniformly.

### T1.2-004 — No response-correlation IDs (single-flight assumption) · **P3** · owned by T1.2-followup
The bridge carries no request ID; the HTML assumes single-flight per response type. Two *different* in-flight requests can't cross (distinct `function` names route to distinct receivers), and AHK's message pump + single-threaded JS serialize handling, so the practical risk is low. But two *same-type* requests (e.g. rapid `loadMoreHistory`) resolve last-writer-wins.
**Fix (optional):** add a monotonic `reqId` echoed in the response envelope; have the HTML ignore stale responses.

### T1.2-005 — Injection surface · **NO ISSUE** (evidence)
- `openUrl` ([settings-ui.ahk:263-266](../../../Development/lib/settings-ui.ahk)): `Run(msg["data"])` is gated by `RegExMatch(msg["data"], "^https://")`. `file://`, `javascript:`, and local-exe paths are rejected. Scheme is validated. ✓
- `viewLogs` (:1234-1258): `Run('explorer.exe "' logDir '"')` where `logDir` is the fixed `A_ScriptDir\data\logs` — no HTML-supplied content. ✓
- `loadLegalDoc` (:1417-1447): `docType` is a **key lookup** into a fixed Map (`privacy`/`terms`/`licenses`), not concatenated into the path → no path traversal. Content is read from bundled local files and injected via `innerHTML`; CSP is set to `default-src 'self'` ([settings-ui.ahk:98](../../../Development/lib/settings-ui.ahk)). ✓
- `deleteHistoryFile`/`clearHistory`, `exportHistory`, `exportConfig`: operate on fixed app paths or user-chosen `FileSelect` save targets. ✓
- `importDictionary`/`importConfig`: `FileSelect` (user picks), JSON-validated + known-key checked, backed up before overwrite (:903-904, :1271-1272). ✓
No action passes HTML string content into an AHK shell/exec path unvalidated.

### T1.2-014 — `exportHistory` lacks CSV formula-injection neutralization · **P3** · owned by T1.2-followup
`HandleExportHistory` quote-escapes fields (`StrReplace(.., '"', '""')`, [settings-ui.ahk:1329-1335](../../../Development/lib/settings-ui.ahk)) but does not neutralize cells beginning with `= + - @`. A transcript starting with `=...` becomes an active formula when the CSV is opened in Excel/Sheets. Self-targeted (user exports own data) → low severity.
**Fix:** prefix risky leading characters with `'` or a space on export.

## Category B — settings ↔ tray IPC robustness

### T1.2-006 — All `0x5555` sends target the correct window · **NO ISSUE** (evidence)
See the [send-site table](#0x5555-quicksay_configreloadmsg-send-sites--all-correct). All five cite `"QuickSay_TrayMode ahk_class AutoHotkey"`.

### T1.2-007 — Tray-not-running silently drops the reload · **P2** · owned by T1.2-followup
Every send is `DetectHiddenWindows(true)` + `if WinExist("QuickSay_TrayMode ...") PostMessage(0x5555,1,0)` (e.g. [settings-ui.ahk:980-982](../../../Development/lib/settings-ui.ahk)). If the tray is dead/crashed (or settings was launched standalone via `QuickSay.exe --settings` with no tray), the `WinExist` guard fails and the reload is **silently skipped**. The config write *does* land on disk (picked up on next tray start), but the user gets no indication that "Apply" had no live effect.
**Fix:** when `WinExist` is false after a save, surface a non-blocking notice ("Changes saved; restart QuickSay to apply") or auto-launch/await the tray.

### T1.2-008 — `HandleImportConfig` writes config.json **without** the config mutex · **P2** · owned by T1.6
Every other config write goes through `SaveJSON`, which holds `AcquireConfigLock` ([settings-ui.ahk:1549-1567](../../../Development/lib/settings-ui.ahk)). `HandleImportConfig` instead does its *own* atomic write — `FileAppend` to `.tmp` then `FileMove` — with **no lock** ([settings-ui.ahk:906-911](../../../Development/lib/settings-ui.ahk)), and reuses the same `config.json.tmp` path `SaveJSON` uses. A concurrent `SaveJSON` (tray writing `lastUpdateCheck`/`lastSeenVersion`) can collide on the temp file or interleave the rename.
**Fix:** route the import through `SaveJSON`/`AcquireConfigLock` like every other write.

### T1.2-009 — Cross-process read-modify-write is not atomic (lost update) · **P2** · owned by T1.6
The mutex serializes only the **write** inside `SaveJSON`, not the surrounding read-modify-write. Handlers do `LoadJSON` (unlocked) → mutate → `SaveJSON` (locked): `HandleSetMode` (:769-773), `HandleSaveModes` (:755-759), `HandleMarkChangelogSeen` (:667-671), `HandleTourCompleted` (:635-642), `HandleClearStartTourFlag` (:646-651). Because `SaveJSON` is a **full-replace** ([settings-ui.ahk:1547-1548](../../../Development/lib/settings-ui.ahk) comment), if the tray reads config, the settings process saves, then the tray writes its (now-stale) copy back (e.g. tray persists `lastUpdateCheck` on the 30 s update check), the settings change is silently discarded — and vice-versa. The mutex prevents a torn file, not a lost update.
**Fix:** hold the lock across the full read-modify-write, or merge-on-write instead of full-replace (load fresh under lock, apply delta, write).
*(Phase-3 note: in the live run the tray did **not** clobber the settings writes — `soundTheme`/`currentMode` persisted correctly — because the tray only *reads* on `0x5555`. The lost-update window opens only when the tray independently *writes* config.)*

### T1.2-010 — Stale static caches in the settings process · **P2** · owned by T1.5 (history) / note
Config-value caches in `SettingsUI`: `static _historyCache` and `static _historyRetention` ([settings-ui.ahk:25-26](../../../Development/lib/settings-ui.ahk)). The settings process is the `0x5555` **sender** and registers **no inbound** config-change listener, so these caches are never invalidated by a tray-side config change. `_historyCache` is invalidated only on clear ([settings-ui.ahk:1200](../../../Development/lib/settings-ui.ahk)) and refreshed only on an `offset=0` load ([settings-ui.ahk:1357](../../../Development/lib/settings-ui.ahk)) — so a transcription made *while the settings window is open* won't appear via "Load More" until the History tab is reloaded. All other statics are UI/handle state (window, WebView, icon handles, hotkey-hook handles), not config values.
**Fix (T1.5):** invalidate `_historyCache` when the tray signals new history (or always re-read on tab focus).

### The clear-history race (T1.2-011) · **P1** · owned by T1.5

> This is the user-reported "history clears, then reappears" bug. P0.2 baseline item 6 marked it **REFUTED — already fixed (commit 2ac7632)**. That conclusion is **incomplete**: the 2ac7632 fix flushes the tray's cache *on reload*, but does not stop an already-scheduled deferred write from rebuilding the file from the stale cache. T1.5 should start from this trace, not re-investigate.

**Airtight trace (line-cited):**

1. **The tray holds the entire history in a persistent in-memory string cache.** `SaveToHistory` lazily loads `HistoryTextCache` from disk on first use and keeps it for the life of the tray process ([QuickSay.ahk:1831-1838](../../../Development/QuickSay.ahk)); after each write it sets `HistoryTextCache := historyText` ([QuickSay.ahk:1882](../../../Development/QuickSay.ahk)). `HistoryCacheLoaded` stays `true`.
2. **History writes are deferred.** When a transcription completes, the tray schedules the write off the message pump: `SetTimer(() => SaveToHistory(_raw, _final, _dur, _audio), -1)` ([QuickSay.ahk:3084](../../../Development/QuickSay.ahk)).
3. **Clearing history (settings process)** deletes the file, invalidates the *settings-side* cache, pushes empty state to the UI, and signals the tray — `FileDelete(historyFile)` + `PostMessage(0x5555,1,0)` ([settings-ui.ahk:1199-1219](../../../Development/lib/settings-ui.ahk)).
4. **The tray's only cache-invalidation path is the deferred reload.** `0x5555` → `SetTimer(ReloadConfig, -100)` ([QuickSay.ahk:215-217](../../../Development/QuickSay.ahk)) → `ReloadConfig` sets `HistoryTextCache := ""`, `HistoryCacheLoaded := false` ([QuickSay.ahk:778-781](../../../Development/QuickSay.ahk)). This is deferred **~100 ms**, and it only clears the cache — it does **not** re-delete the file.

**The race:** the deferred `SaveToHistory` (step 2, fires ~immediately) and `ReloadConfig` (step 4, fires ~100 ms later) are both timer callbacks competing on the same single-threaded pump. If a `SaveToHistory` runs *after* the file was deleted but *before* `ReloadConfig` clears the cache — which is the likely ordering, since the write timer is `-1` and the reload is `-100`, and the tray's pump may be blocked during the synchronous Whisper/LLM HTTP calls — then `SaveToHistory` rebuilds `historyText` from the still-populated `HistoryTextCache` (all pre-clear entries + the new one) and `AtomicWriteFile`s it back ([QuickSay.ahk:1841-1882](../../../Development/QuickSay.ahk)). Every "deleted" entry reappears on disk, and nothing deletes it again. `ReloadConfig` then clears the in-memory cache — too late; the file is already rewritten.

**Reproduction recipe (3-4 steps — T1.5 inherits verbatim):**
1. Start QuickSay (tray) and do **2-3 dictations** so `HistoryTextCache` is populated in memory and `data/history.json` has entries.
2. Trigger one more dictation and, **while it is still processing** (overlay showing; before the entry lands), open Settings → History (or Processing) tab and click **Clear History → confirm**.
3. Wait for the in-flight transcription's deferred `SaveToHistory` to fire (a second or two), then click **Refresh** on the History tab.
4. **Observed:** the just-cleared entries reappear (the in-flight write rebuilt the file from the stale cache). **Expected:** history stays empty.
   *Deterministic variant:* because the cache persists across transcriptions, the race also fires if step 2's clear is immediately followed by any new dictation before `ReloadConfig` lands.

**Fix direction (T1.5, not this session):** make the clear authoritative against the cache — e.g. a dedicated "history cleared" message that flushes `HistoryTextCache` synchronously *and* sets a generation/`clearedAt` guard so any deferred `SaveToHistory` scheduled before the clear is dropped (or re-reads the now-empty file instead of trusting the cache).

## Category C — History pagination & correctness

### T1.2-012 — `getHistoryCount` over-counts by >2× (line-count vs entry-count) · **P2** · owned by T1.5
`HandleGetHistoryCount` counts physical lines: `Loop Read, this.historyFile { count++ }` ([settings-ui.ahk:1190-1197](../../../Development/lib/settings-ui.ahk)). But `history.json` is a **pretty-printed JSON array** where `SaveToHistory` writes each entry on its own line **separated by blank indent lines** ([QuickSay.ahk:1844-1851](../../../Development/QuickSay.ahk)).
**Live evidence (Phase 3):** `data/history.json` with **10** entries (`"id"` count = 10) has **20 newlines / ~21 logical lines**. So the Processing tab renders **"21 entries saved"** ([settings.html:1394](../../../Development/gui/settings.html)) while the History tab — which parses the array and uses `data.Length` ([settings-ui.ahk:1360-1386](../../../Development/lib/settings-ui.ahk), [settings.html:2242-2254](../../../Development/gui/settings.html)) — correctly shows **"10 entries."** The two tabs disagree for the same file.
**Fix:** count parsed array entries (`LoadJSON(...).Length`), not file lines.

### T1.2-013 — Large history blocks the UI on a full parse-then-slice · **P2** · owned by T1.5
On a fresh load (`offset=0`), `HandleLoadHistoryData` reads and `JSON.Parse`s the **entire** `history.json`, then truncates to `retention` and caches it ([settings-ui.ahk:1357-1380](../../../Development/lib/settings-ui.ahk)). Pagination then slices the in-memory array. So the page size (100) limits the *render*, not the *parse* — the full file is parsed synchronously on the AHK message thread on every cold load. With a multi-thousand-entry file this stalls the settings UI on open/refresh.
**Fix (T1.5):** cap the parse (retention is enforced tray-side anyway), or stream/lazily parse.

### T1.2-015 — Pagination boundaries, empty/error states · **NO ISSUE** (evidence)
- Page math: `pageSize=100`, `endIdx=Min(offset+pageSize,total)`, 1-based `idx=offset+1..endIdx`, `hasMore=(offset+pageSize)<total` ([settings-ui.ahk:1385-1398](../../../Development/lib/settings-ui.ahk)). HTML appends with `historyData.concat` on `offset>0`, replaces on `offset===0` ([settings.html:2245-2250](../../../Development/gui/settings.html)). No dropped or duplicated entry at a page boundary.
- `LoadJSON` returns `Map()` on missing/empty/parse-error ([settings-ui.ahk:1533-1545](../../../Development/lib/settings-ui.ahk)); history code coerces a non-array to `[]` ([settings-ui.ahk:1361-1363](../../../Development/lib/settings-ui.ahk)); HTML renders an empty-state card for `length===0` ([settings.html:2302-2306](../../../Development/gui/settings.html)). Corrupted/absent `history.json` → empty list, no crash.

### T1.2-016 — Retention `0` ("unlimited") + trim removes the **newest** entries · **P1** · owned by T1.5 (cross-ref, tray-side)
The UI clamps `historyRetention` to `{0, 100, 500}` ([settings.html:1586](../../../Development/gui/settings.html)) and `0` is offered as "unlimited." That `0` passes straight through the engine's int normalization (`if cfg.Has("historyRetention") → Integer(...)`, [QuickSay.ahk:1657-1664](../../../Development/QuickSay.ahk)) to `Config["history_retention"] = 0`. In `SaveToHistory`, `maxHistory := 0` makes `entryCount > maxHistory` true for any non-empty history, so the trim block runs ([QuickSay.ahk:1854-1877](../../../Development/QuickSay.ahk)). It is saved from wiping everything only by an off-by-one (the loop can match `entryCount-1` separators, so `trimmed = trimCount` fails and the rebuild is skipped) — i.e. "unlimited" works **by accident.** Separately, the trim removes entries **from the front** of a **newest-first** array (`SubStr(historyText, trimPos+1)` after advancing past leading `},` separators) — so when history *does* exceed a finite limit, it discards the **newest** entries instead of the oldest.
**Cross-ref:** tray-side `SaveToHistory`; surfaced here because the UI emits the `0` value. T1.5 owns the fix; flagging so T1.5 starts from evidence.

- **Doc note:** CLAUDE.md's stats schema list (`byApp, byDay, dailyStreak, averageWPM, totalDuration`) is **incomplete** — it omits `totalWords`, `totalSessions`, `wpmWords`, `wpmDuration`, etc. Harmless, but worth updating. *(Streak/DST math is computed tray-side in `UpdateStatistics` — out of scope; flagged to T1.1 if it audits stat computation.)*

### T1.2-018 — No dark/light toggle; fixed dark theme only · **P2** · owned by T1.7
There is no theme toggle, no `prefers-color-scheme`, no `data-theme`/`.dark` class anywhere in `settings.css` or `settings.html` (grep: zero matches). ### T1.2-019 — Sub-label text fails WCAG AA contrast · **P2** · owned by T1.7
**Measured live** (computed `:root` token values; WCAG relative-luminance ratios):

| Pair | Colors | Ratio | AA normal (4.5:1) |

`--text-tertiary` ([settings.css:31](../../../Development/gui/settings.css)) is used for hints, `.sub-text`, and history meta lines (e.g. "5 min ago · claude", [settings.html:2321-2327](../../../Development/gui/settings.html)) — real content that fails AA. `--text-quaternary` ([settings.css:32](../../../Development/gui/settings.css)) fails badly; confirm it is only ever used for decorative/disabled text (e.g. the italic tagline at [settings.html:716](../../../Development/gui/settings.html)).
**Fix (T1.7):** lighten `--text-tertiary` to ≥ #8b8b9e-ish (≈4.5:1 on `--bg-base`); audit every `--text-quaternary` use for readable content.

### T1.2-020 — No responsive breakpoints; fixed layout clips below ~700px · **P2** · owned by T1.7
`settings.css` contains **no `@media` queries at all** (grep: zero). Layout is fixed; `body { overflow:hidden; position:fixed }` ([settings.css:57-69](../../../Development/gui/settings.css)) means content that doesn't fit is **clipped, not scrollable**. The window enforces a 700×600 minimum via `WM_GETMINMAXINFO` ([settings-ui.ahk:179-185](../../../Development/lib/settings-ui.ahk)).
**Live evidence (Phase 3 screenshots, `tests/playwright/artifacts/`):**
- `t12-700x600.png` (at the enforced minimum): renders **cleanly** — sidebar, content sections, sticky action bar all fit.
- `t12-540x460.png` (below the minimum, via CDP emulation): content cards clip on the right with **no horizontal scrollbar** (`document.documentElement.scrollWidth === window.innerWidth`, `hOverflow=false`) → clipped controls are unreachable.
Because 700×600 is enforced, this is a **latent** risk (reachable only via high-DPI scaling shrinking the effective CSS viewport, or a future lowering of the minimum), not an active break at supported sizes.
**Fix (T1.7):** add breakpoints for sidebar collapse / content reflow, or allow an inner scroll container so sub-minimum sizes degrade gracefully.

### T1.2-021 — Custom toggles may be keyboard-unreachable · **P2** (needs confirmation) · owned by T1.7
The toggle/checkbox controls render a styled switch over a hidden raw `<input>`. In Phase 3, Playwright reported `#autoPaste` and `#btnChangeHotkey` as **"element is not visible"** (the latter because `customHotkeyControls` is `display:none` while the default hotkey is active — expected). For the checkboxes, if the raw `<input>` is hidden via `display:none`/`visibility:hidden` it is removed from tab order and cannot be toggled by keyboard. (Playwright "not visible" can also mean `opacity:0`, which *is* still focusable — hence "needs confirmation.") `Ctrl+S` to save works ([settings.html:3200-3203](../../../Development/gui/settings.html)).
**Fix (T1.7):** verify each interactive control is keyboard-focusable with a visible focus ring; if inputs are visually hidden, use the `opacity:0`/`sr-only` pattern (focusable) rather than `display:none`.

### T1.2-022 — Guided tour · **NO ISSUE** (evidence)
9-step tour ([settings.html:3230-3240](../../../Development/gui/settings.html)). All seven element targets exist (verified by grep): `#stickyModeToggle` (1), `.hotkey-options` (1), `.section` (audio), `#modeList` (1), `#dictionary h2`, `.filter-bar` (1), `.stats-grid`. Missing targets are handled gracefully — `showTourStep` skips to the next step or ends ([settings.html:3294-3298](../../../Development/gui/settings.html)). Completion persists correctly: `endGuidedTour` updates local config and fires `tourCompleted` ([settings.html:3613-3626](../../../Development/gui/settings.html)) → `HandleTourCompleted` sets `tourCompleted=true`, `showGuidedTour=false`, deletes `startTourOnOpen` ([settings-ui.ahk:634-643](../../../Development/lib/settings-ui.ahk)). Works in the single dark theme. *(Small-viewport spotlight positioning uses `getBoundingClientRect` math, [settings.html:3303-3333](../../../Development/gui/settings.html) — could mis-place at the 700×600 minimum; minor, T1.7 may spot-check.)*
