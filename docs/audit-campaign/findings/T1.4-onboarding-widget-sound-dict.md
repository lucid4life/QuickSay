### The six required verdicts (up front)

1. **Accidental-recording-on-drag fix — HELD.** The 5px threshold is intact at [widget-overlay.ahk:341](../../../Development/widget-overlay.ahk); a move >5px flips `wasClick→false`/`isDragging→true`, and mouseup only toggles recording when `wasClick` is still true. Confirmed by code trace. *(The live 2px-nudge / 10px-drag test was not run this session — see Phase 3.)*
2. **Mouse-capture release — VERIFIED, NOT a P0.** `ReleaseCapture` runs unconditionally on mouseup *before* the recording toggle ([widget-overlay.ahk:309](../../../Development/widget-overlay.ahk)) and again in `Hide()` ([:152-155](../../../Development/widget-overlay.ahk)). A throwing `StartRecording()`/`StopAndProcess()` cannot strand capture because release precedes it. Residual: no `WM_CAPTURECHANGED` handler (T1.4-014, P3) — but that only leaves a stale *flag*, never a stuck OS capture, so the desktop cannot freeze.
3. **18 sound files — ALL PRESENT + VALID.** 6 themes × 3 events, every file non-zero with a valid `RIFF…WAVE` header (table in T1.4-017). No missing/corrupt file. (Audibility inferred from valid headers + sizes; playback not performed autonomously.)
4. **Dictionary regex-escape — SAFE (escaped).** User "spoken" terms are regex-escaped at [QuickSay.ahk:2040](../../../Development/QuickSay.ahk) before being joined into the alternation; the isolated test proved `c++`→`c\+\+`, `a.*b` stays literal, and metacharacter-only keys (`(((`, `(.*)+`) neither throw nor hang. **No ReDoS, no injection.** (The real dictionary issue is correctness, not security — T1.4-023.)
5. **Onboarding API-key DPAPI storage — CONFIRMED.** The key is DPAPI-encrypted before write when it starts with `gsk_` ([onboarding_ui.ahk:352-360](../../../Development/onboarding_ui.ahk)); on encrypt failure it is **not** saved. It is never written to the debug log (only `err.Message` is). One dead plaintext fallback branch exists but is unreachable in the live flow (T1.4-006, P3).
6. **a11y / multi-monitor / UX-jargon →** tagged `owned by T1.7` where applicable. Notably the onboarding HTML already *fixes* the T1.2 contrast finding for its own surface (`--text-tertiary` overridden to `#a0a0b8`, [onboarding.html:14](../../../Development/gui/onboarding.html)) — a positive, not a finding.

### Top findings in plain English

1. **"Learn from Selection" lies about success** (T1.4-025, **P1**). The dictionary-learning flow saves the new word and shows *"Added N correction(s)"*, but it never recompiles the match pattern — so the learned correction does **nothing** to subsequent transcriptions until the user next saves a setting (which fires a `0x5555` reload) or restarts the app.
2. **The "default" sound theme is a build-time landmine** (T1.4-019, P2). The shipped default config points `sound_theme` at `"default"`, which resolves to top-level `sounds\start.wav` etc. Those files exist in the *installed* app but are **absent from the `Development/` source tree**, and `setup.iss` bundles `sounds\*` recursively — so the **next installer built from the current tree would ship no default sounds**, silently downgrading every fresh user to raw `SoundBeep` tones.
3. **`mechanical` is installed but unselectable** (T1.4-017, P2 — verifies T1.2-024). The `mechanical/` theme has 3 valid WAVs on disk, but the settings `#soundTheme` dropdown lists `default, subtle, click, crystal, neon, bloom, silent` — `mechanical` is missing, while `default`/`silent` are pseudo-themes with no directory.

## Category A — Onboarding wizard

### T1.4-001 — Step navigation & gating · **NO ISSUE** (evidence)
No dead-end: every step ≤4 has a working Next; Back is shown for idx>0 ([onboarding.html:1431](../../../Development/gui/onboarding.html)) and panel state/JS vars persist across navigation. Step 1 cannot be passed without a *tested* key (`apiKeyValid`) — empty key and untested key both produce inline errors and block ([:1475-1487](../../../Development/gui/onboarding.html)). Because the Done step (5) is only reachable by passing step 1's gate (or skipping, which closes the window rather than advancing), **the wizard cannot complete without a validated key** — closing the relevant idempotency hole. The mic and try-dictation steps are intentionally skippable.

### T1.4-002 — "Skip for now" depends solely on `window.close()` with no `WindowCloseRequested` handler · **P2** · owned by T1.4-followup *(needs live confirmation)*
`skipSetup()` confirms, then calls `window.close()` only ([onboarding.html:1517-1522](../../../Development/gui/onboarding.html)) — it posts **no** action to AHK. `onboarding_ui.ahk` registers only `add_WebMessageReceived` ([:110](../../../Development/onboarding_ui.ahk)); there is **no `add_WindowCloseRequested`** handler. In WebView2, script `window.close()` raises `WindowCloseRequested`, and with no host handler the host window is **not** closed by default — so "Skip for now" likely does nothing (the user is stuck on the wizard, with the title-bar X as the only escape, which *does* work via `OnClose`). Contrast `finishOnboarding`/`finishAndStartTour`, which first `postToAHK('finish'…)` (AHK destroys the GUI + `ExitApp`) and only use `window.close()` as a redundant fallback ([:1549-1570](../../../Development/gui/onboarding.html)) — those work.
**Fix:** route `skipSetup` through a `postToAHK('skip', …)` action that closes the window AHK-side (or add an `add_WindowCloseRequested` handler).

### T1.4-003 — API-key test error display · **NO ISSUE / GOOD** (Dad-Test pass)
`apiTestResult` maps HTTP status to plain English with **no jargon** ([onboarding.html:1798-1808](../../../Development/gui/onboarding.html)): 401→"Invalid API key…", 0→"Could not connect. Please check your internet connection.", 429→"Too many requests…", ≥500→"Groq service is temporarily unavailable…". Only the generic *else* leaks a code ("Connection failed (HTTP {n})…"), and even that has a plain-English suffix. Meets the CLAUDE.md "Dad Test." (The test does a real network round-trip to `/v1/models`, not a format check — [onboarding_ui.ahk:276-281](../../../Development/onboarding_ui.ahk).)

### T1.4-004 — Mic test exercises the default device only; never the named-device (FFmpeg) path · **P2** · owned by T1.4-followup
Both the recording (`HandleTestRecording5s`/`HandleStartTestTranscription`, MCI `waveaudio`, [onboarding_ui.ahk:546-547,606-607](../../../Development/onboarding_ui.ahk)) and the level meter (WASAPI `eCapture/eConsole` default device, [:495-497](../../../Development/onboarding_ui.ahk)) use the **Windows default capture device**. Per CLAUDE.md, MCI captures only the default device — named devices require the FFmpeg `-f dshow` path at runtime. So a user whose configured `audioDevice` is a named device (the `micDeviceName` panel even *displays* that name via `HandleGetMicDevice`, [:529-537](../../../Development/onboarding_ui.ahk)) has the mic test silently measure a **different** device than the app will record from → false confidence (pass here, broken at runtime, or vice-versa). Low frequency at first-run (default config sets `audioDevice:"Default"`), hence P2 not P1.
**Fix:** when `audioDevice` is non-Default, test via the same FFmpeg path the engine uses; otherwise label the test "default device."

### T1.4-005 — No-mic / muted / busy-mic feedback · **NO ISSUE / GOOD** (evidence)
MCI return codes are not checked, but the outcome is covered downstream: `FinishTestRecording` requires `FileGetSize > 1000` ([onboarding_ui.ahk:583](../../../Development/onboarding_ui.ahk)), and `analyzeTestResults` branches on collected peak samples — `maxPeak < 0.03` → *"No audio detected"* with an actionable checklist (plugged in / not muted / correct device / other apps / try another mic), [onboarding.html:1667-1680](../../../Development/gui/onboarding.html). Low signal triggers up to 3 automatic volume boosts before advising. Graceful, no hang, Dad-Test-grade advice.

### T1.4-006 — Dead plaintext-key fallback branch · **P3** · owned by T1.4-followup
`HandleSaveKey` DPAPI-encrypts when the key starts with `gsk_`, else stores it **plaintext** (`cfg["groqApiKey"] := apiKey`, [onboarding_ui.ahk:365-367](../../../Development/onboarding_ui.ahk)). In the live flow `HandleSaveKey` is only reached from `HandleTestAPI` after a 200 from Groq (so the key is a real `gsk_` key) — the `saveKey` action that would pass an arbitrary key is never sent (T1.4-007). The branch is therefore unreachable today, but it is a latent plaintext-storage path.
**Fix:** drop the plaintext branch; refuse to persist a key that fails to encrypt.

### T1.4-007 — Bridge: no `default` case for unknown actions; one dead action · **P3** · owned by T1.4-followup
The dispatcher is an if/else chain with **no else/`default`** ([onboarding_ui.ahk:229-261](../../../Development/onboarding_ui.ahk)) — an unknown action is silently dropped (not even `OutputDebug`-logged, unlike T1.2-003). Separately, the JS `saveAPIKey()`→`saveKey` path ([onboarding.html:1545-1547](../../../Development/gui/onboarding.html)) is **defined but never called** (the key is saved AHK-side inside `HandleTestAPI`), so the `saveKey` action is dead. Outer `try/catch` swallows any handler throw to a log file ([onboarding_ui.ahk:262-264](../../../Development/onboarding_ui.ahk)).
**Fix:** add an else branch that logs unknown actions; remove the dead `saveAPIKey`/`saveKey` path.

### T1.4-008 — `openURL` runs unvalidated input (`Run(data)`) · **P3** · owned by T1.4-followup
`openURL` → `try Run(data)` with **no scheme check** ([onboarding_ui.ahk:259-261](../../../Development/onboarding_ui.ahk)), unlike settings' `openUrl` which is gated by `^https://` (T1.2-005). The only caller passes a hardcoded `https://console.groq.com/keys` ([onboarding.html:1092](../../../Development/gui/onboarding.html)) and the page is local + CSP-restricted, so exploitation requires already-tampered local files — low practical risk, but an inconsistent, unhardened sink.
**Fix:** gate with `^https?://` to match the settings handler.

### T1.4-009 — API error rendered via `innerHTML`; looser onboarding CSP · **P3** · owned by T1.4-followup / T1.7
The transcribed text is rendered safely with `textContent` ([onboarding.html:1869](../../../Development/gui/onboarding.html)), but the API **error** string is injected with `innerHTML`: `status.innerHTML = svgX + ' Error: ' + msg.error` ([:1863](../../../Development/gui/onboarding.html)). `msg.error` is the Groq response `message` field (or an AHK `err.Message`), JSON-escaped but **not HTML-escaped** in `SendTranscriptionError` ([onboarding_ui.ahk:707-713](../../../Development/onboarding_ui.ahk)). The onboarding CSP also allows `script-src 'self' 'unsafe-inline'` ([:114](../../../Development/onboarding_ui.ahk)) — looser than settings' `default-src 'self'`. Source (Groq) is trusted, so this is defense-in-depth, not an active hole.
**Fix:** render the error with `textContent`; drop `'unsafe-inline'` from `script-src`.

### T1.4-010 — Completion + idempotency · **NO ISSUE** (evidence)
`MarkOnboardingDone` writes the `onboarding_done` marker ([onboarding_ui.ahk:438-448](../../../Development/onboarding_ui.ahk)). The X-button path (`OnClose`) marks done **only** with a valid/encrypted key, so abandoning mid-wizard re-fires next launch ([:170-187](../../../Development/onboarding_ui.ahk)). `FinishAndExit` (reached only from the Done step, i.e. past the key gate) always marks done ([:394-436](../../../Development/onboarding_ui.ahk)). Manual re-run works — `Show()` builds fresh and does not consult the marker (the *decision to launch* the wizard lives tray-side in `QuickSay.ahk` — boundary noted for T1.1). COM is released before `Destroy` on both exit paths ([:203-208,427-431](../../../Development/onboarding_ui.ahk)), so msedgewebview2 terminates cleanly.

## Category B — Widget drag/click edge cases

### T1.4-011 — Accidental-recording-on-drag (the prior fix) · **HELD — NO REGRESSION** (evidence)
The 5px threshold is intact at [widget-overlay.ahk:341](../../../Development/widget-overlay.ahk): `if (this.wasClick && (Abs(dx) > 5 || Abs(dy) > 5)) { wasClick:=false; isDragging:=true }`. Mouseup toggles recording only `if (this.wasClick)` ([:312-318](../../../Development/widget-overlay.ahk)); once a drag is detected, `wasClick` is false so no recording fires — instead `SavePosition` runs. Logic is sound. *(Live 2px/10px confirmation pending — Phase 3.)*

### T1.4-012 — Drag threshold is physical-pixel, not DPI-aware · **P3** · owned by T1.7
`MouseGetPos` under `CoordMode("Mouse","Screen")` returns physical screen pixels, so the 5px threshold is a fixed physical distance ([widget-overlay.ahk:335-341](../../../Development/widget-overlay.ahk)). On a 200 %% display, 5 physical px is a smaller hand-movement, so a jittery intended *click* more easily crosses into "drag" — in which case mouseup takes the `isDragging` branch and **does not toggle recording** (a missed click, the inverse of the original bug). Minor; document or scale the threshold by `A_ScreenDPI`.

### T1.4-013 — Position persistence / off-screen recovery · **NO ISSUE** (evidence)
`Show()` loops monitors and resets to the default right-edge-center if the saved point isn't on any current monitor ([widget-overlay.ahk:54-94](../../../Development/widget-overlay.ahk)); `SavePosition` clamps to the work area of the monitor containing the widget center (else primary) before writing ([:515-538](../../../Development/widget-overlay.ahk)); the drag path also clamps live ([:351-369](../../../Development/widget-overlay.ahk)). Monitor disconnect / resolution change → widget reappears at the default, never vanishes. (Multi-monitor polish is T1.7's domain; no defect here.)

### T1.4-014 — Mouse-capture hygiene · **VERIFIED (not P0); residual P3** · owned by T1.4-followup
Capture is released on every normal exit: mouseup (`ReleaseCapture` at [:309](../../../Development/widget-overlay.ahk), **before** the recording toggle) and `Hide()` ([:152-155](../../../Development/widget-overlay.ahk)). Because release precedes `StartRecording()/StopAndProcess()`, a throw in those cannot strand capture → **no desktop-freeze P0**. Residual: there is **no `WM_CAPTURECHANGED` (0x215) handler**, so if the OS revokes capture mid-drag, `_captured` stays stale-`true` until the next mousedown/up corrects it — a stale flag, not a stuck OS capture.
**Fix (minor):** handle `WM_CAPTURECHANGED` to clear `_captured`/`isDragging`.

### T1.4-015 — Focus stealing · **NO ISSUE** (evidence)
Window is `+E0x08000000` (`WS_EX_NOACTIVATE`) + shown `NoActivate` ([widget-overlay.ahk:99-101](../../../Development/widget-overlay.ahk)), and `WM_MOUSEACTIVATE` returns `MA_NOACTIVATE` (3) ([:473-477](../../../Development/widget-overlay.ahk)). Clicking the widget cannot steal focus from the target app, so the paste lands in the user's app, not the widget owner. The drag uses manual `SetCapture` + `gui.Move` specifically to avoid the system modal move-loop that would steal focus ([:299](../../../Development/widget-overlay.ahk)).

### T1.4-016 — Widget destroy hygiene + `HideAndUpdateConfig` writes config without the mutex · **P2** · owned by T1.6 (+ P3 housekeeping)
`Hide()` cleanly stops timers, releases capture, destroys the tooltip + GUI, and unregisters all six `OnMessage` hooks ([widget-overlay.ahk:131-184](../../../Development/widget-overlay.ahk)) — no leaked GDI/window handle. **Housekeeping (P3):** it nulls five `_bound*` refs but not `_boundMouseLeave` ([:170-174](../../../Development/widget-overlay.ahk)) — benign (re-`Show()` reassigns it). **P2:** the right-click "Hide Widget" path, `HideAndUpdateConfig`, rewrites the whole `config.json` (atomic tmp+rename) with **no `AcquireConfigLock`** ([:456-471](../../../Development/widget-overlay.ahk)) — same class as T1.2-008. A concurrent settings-process `SaveJSON` can lose-update or collide on the temp path. (Note `SavePosition` *does* take the lock — [:547](../../../Development/widget-overlay.ahk) — so the omission here is inconsistent.)
**Fix:** route `HideAndUpdateConfig` through the locked write path; null `_boundMouseLeave` for symmetry.

### T1.4-019 — `default` theme + missing top-level WAVs in the source tree = build-time regression · **P2** · owned by T1.4-followup (+ cross-ref T1.3 build inputs)
Default config sets `sound_theme := "default"` ([QuickSay.ahk:1287](../../../Development/QuickSay.ahk); normalization default also `"default"` at [:1599](../../../Development/QuickSay.ahk)). With `theme=="default"`, `PlaySound` skips the themed branch and looks for top-level `sounds\<soundType>.wav` ([:1724-1729](../../../Development/QuickSay.ahk)). **The installed app has those** (`sounds\start.wav`, `stop.wav`, `error.wav` confirmed under `%LOCALAPPDATA%\Programs\QuickSay Beta\sounds\`), so production default-theme audio works today. **But the `Development/` source tree has no top-level `sounds\*.wav`**, and `setup.iss:88` bundles `Source: "sounds\*" … recursesubdirs` — so an installer **built from the current tree would ship no default sounds**, silently downgrading every fresh user (default theme) to raw `SoundBeep`. `PreWarm` even guards `if FileExist(SoundsDir "\start.wav")` ([:3314-3315](../../../Development/QuickSay.ahk)) — a no-op in dev because the file is absent.
**Fix:** change the default `sound_theme` to a real installed theme (e.g. `subtle` or `click`) so the default no longer depends on un-tracked top-level files; or add/track `sounds\{start,stop,error}.wav` in `Development/`.

### T1.4-020 — `playSounds` gate + theme switch + `previewSound` · **NO ISSUE** (evidence) + minor note
Gate works: `sounds_enabled=0` (from `playSounds`) returns immediately ([QuickSay.ahk:1707-1708](../../../Development/QuickSay.ahk)); `silent` returns early ([:1712-1713](../../../Development/QuickSay.ahk)). Theme switch takes effect without restart via the `0x5555` reload (`ReloadConfig`→`LoadConfig`, [:778-784](../../../Development/QuickSay.ahk)), which `saveConfig` fires (T1.2 matrix #2). `previewSound`→`HandlePreviewSound` resolves identically and falls back to `SoundBeep(600,100)` ([settings-ui.ahk:674-699](../../../Development/lib/settings-ui.ahk)) — previewing "default" emits three beeps, consistent with T1.4-019. **Minor (P3-adjacent):** the preview uses blocking `Sleep(600)` between sounds ([settings-ui.ahk:696-697](../../../Development/lib/settings-ui.ahk)) — it stalls the *settings* message pump ~1.2 s, but is off the recording hot path; not flagged as a separate issue.

### T1.4-021 — No blocking play on the hot path · **NO ISSUE** (evidence)
`SoundPlay(file)` is called without the wait flag → asynchronous; the `SoundBeep` fallbacks are explicitly deferred via `SetTimer(…,-1)` "so they don't block processing" ([QuickSay.ahk:1731-1741](../../../Development/QuickSay.ahk)). The record-start sound ([:2736](../../../Development/QuickSay.ahk)) therefore does not sit synchronously between hotkey-press and recording-start.

## Category D — Dictionary regex safety + round-trip

### T1.4-022 — ReDoS / injection · **SAFE — terms are escaped** (verified) — headline verdict
`CompileDictionaryPattern` escapes every user "spoken" term before joining: `escapedKey := RegExReplace(key, "[.*+?^${}()|[\]\\]", "\$0")` ([QuickSay.ahk:2040](../../../Development/QuickSay.ahk)), then builds `"i)\b(" StrJoin(escaped,"|") ")\b"` ([:2046](../../../Development/QuickSay.ahk)). The escape class covers every PCRE metacharacter that is special in `i)` (non-extended) mode. **Isolated executable proof** (replica of the compile+apply logic, run under AutoHotkey64.exe):

```
T1 pattern        = i)\b(c\+\+)\b
T3 pattern        = i)\b(a\.\*b)\b
T3 'a.*b and aXb' = 'LIT and aXb'      ← a.*b stays literal; aXb NOT matched (no wildcard)
T5 metachar keys  = no throw, no hang  ← keys '(((', '((((((((((', '(.*)+' escaped fine
```

Because the alternation is composed only of escaped literals (no nested quantifiers, no back-references), there is **no catastrophic backtracking** — matching is ~O(text_length × n_entries) worst case (PCRE's first-char bitmap skips most positions), so even thousands of entries match in low-millisecond time. **No ReDoS, no injection vector.** A crafted dictionary entry cannot break the compile or hang the hot path.

### T1.4-023 — `\b…\b` anchoring silently kills entries with leading/trailing non-word chars · **P2** · owned by T1.4-followup
The word-boundary anchors mean any term whose first or last character is a non-word character (`+ # . @`, accented letters under default ASCII `\b`) never matches. Verified in the same isolated run:

```
T1 'love c++ ok'  = 'love c++ ok'      (UNCHANGED — trailing \b fails after '+')
T7 '.net and c#'  = 'I use .net and c# daily'  (UNCHANGED)
```

For a dictation app with a dedicated **Code** mode, programmer terms (`c++`, `c#`, `.net`, `node.js`) are exactly the entries a user would add — and they silently do nothing. (Practical incidence is dampened because Whisper usually emits *words* like "c plus plus" on the spoken side, which match fine; the failure bites literal-symbol spoken terms.) Multi-word phrases and ordinary words work (`'so for example yes' → 'so e.g. yes'`).
**Fix:** drop the unconditional `\b…\b`; use boundary assertions that are conditional on the term's edge characters (e.g. only require `\b` where the adjacent term char is a word char), or switch to `(*UCP)` for Unicode word handling.

### T1.4-024 — `ParseDictionaryJson` is a brittle regex parser, not `JSON.Parse` · **P2** · owned by T1.4-followup
`LoadDictionary` parses `dictionary.json` with two `RegExMatch` patterns, not the JSON library ([QuickSay.ahk:1675-1698](../../../Development/QuickSay.ahk)): Format-1 `'"spoken"\s*:\s*"([^"]+)"\s*,\s*"written"\s*:\s*"([^"]+)"'` and Format-2 `'"corrections":\s*\{([^}]+)\}'`. Consequences: (a) `[^"]+` means any value containing an escaped quote (`\"`), and escapes like `\n`/`\\`/`\uXXXX`, are mis-captured or taken literally → **not lossless**; (b) it is order-dependent (`spoken` must immediately precede `written`); (c) the legacy block breaks on any `}` inside it. The *write* paths use proper JSON (`AddToDictionary` via `EscapeJSON`, [:2682](../../../Development/QuickSay.ahk); the settings import via the JSON lib), so the **read parser is asymmetric to the writers** — a value with a quote round-trips lossily. Upside: a malformed file degrades gracefully — `LoadDictionary`'s `try/catch` falls back to `Map()` ([:1542-1547](../../../Development/QuickSay.ahk)); **no crash** (verified by reading the guard). 
**Fix:** parse with `JSON.Parse` (the lib is already loaded) and iterate the array/object.

### T1.4-025 — `AddToDictionary` reports success but never recompiles → learned word is inert until reload · **P1** · owned by T1.4-followup
`LearnFromSelection` ("Learn from Selection") shows *"Added N correction(s) to dictionary"* ([QuickSay.ahk:2580](../../../Development/QuickSay.ahk)) when `AddToDictionary` returns true. `AddToDictionary` mutates the in-memory `Dictionary` ([:2659](../../../Development/QuickSay.ahk)) and rewrites the file, **but never calls `CompileDictionaryPattern`** — so `DictCompiledPattern` and `DictReplacements` (the structures `ApplyDictionary` actually uses, [:2011-2016](../../../Development/QuickSay.ahk)) stay stale. The newly-learned correction therefore has **zero effect** on subsequent transcriptions until something calls `LoadDictionary` again — i.e. the next `0x5555` config reload (`ReloadConfig`→`LoadDictionary`, [:783](../../../Development/QuickSay.ahk)) or an app restart. The UI's success message is misleading; the advertised feature silently no-ops in-session.
**Fix:** call `CompileDictionaryPattern()` at the end of `AddToDictionary` (or once after the `LearnFromSelection` loop).

### T1.4-026 — `AddToDictionary` writes `dictionary.json` without the config mutex · **P2** · owned by T1.6
`AddToDictionary` writes via `AtomicWriteFile` ([QuickSay.ahk:2687](../../../Development/QuickSay.ahk)) but **without `AcquireConfigLock`**. The settings-side dictionary save *does* hold the lock (`SaveJSON`, P0.2 item 4 / [settings-ui.ahk:1549](../../../Development/lib/settings-ui.ahk)). So a tray-side "Learn from Selection" concurrent with a settings-tab Save/Import races: the atomic rename prevents a torn file, but a lost update (last-writer-wins) can drop one side's change. Same concurrency family as T1.2-008 / the importConfig finding.
**Fix:** acquire `AcquireConfigLock` around the read-modify-write in `AddToDictionary`.

### T1.4-027 — Empty / disabled dictionary · **NO ISSUE** (evidence)
`ApplyDictionary` short-circuits when `dictionary_enabled` is false/absent ([QuickSay.ahk:2000-2001](../../../Development/QuickSay.ahk)) and again when `DictCompiledPattern==""` or `Dictionary.Count==0` ([:2003-2004](../../../Development/QuickSay.ahk)). `CompileDictionaryPattern` returns early on an empty dictionary, leaving `DictCompiledPattern := ""` ([:2031-2035](../../../Development/QuickSay.ahk)). So although the isolated test showed a *naive* compile would build `i)\b()\b` (empty group), the real code's early-return guard means that broken pattern is **never built or used**. No crash on empty/disabled.
