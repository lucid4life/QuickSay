## Category A ‚Äî Installer file integrity

### T1.3-001 ‚Äî Installer ships the developer's live `config.json` (encrypted key + personal prefs) ‚Äî **P1**
**File:** `setup.iss:72` (`Source: "config.json"; ‚Ä¶ Flags: onlyifdoesntexist`) ‚Üí ships `Development/config.json`.
**Evidence:** `config.json` on disk is the dev's working config:
```json
"groqApiKey": "AQAAANCMnd8BFdERjHoAwE/Cl+sBAAAA‚Ä¶",   // DPAPI blob
"launchAtStartup": 1, "widgetX": 405, "widgetY": 705,
"lastWeeklySummary": "20260513002031", "lastUpdateCheck": "2026-05-19",
"historyRetention": 500, "lastSeenVersion": "1.9.0-beta", "soundTheme": "default"
```
On a **fresh install** (`onlyifdoesntexist` ‚áí dropped because the user has none yet) the new user inherits all of these. Three concrete harms, in severity order:

1. **Confirmed first-run break ‚Äî the onboarding completion gate is defeated (root cause).** `onboarding_ui.ahk OnClose()` (lines 170-187) decides whether to write the `onboarding_done` marker based on whether a key "looks set":
   ```ahk
   if (SubStr(key, 1, 4) == "gsk_" && StrLen(key) >= 20)        ; real plaintext key
       hasKey := true
   else if (SubStr(key, 1, 4) != "gsk_" && StrLen(key) >= 50)   ; "looks like an encrypted blob"
       hasKey := true
   if (hasKey) this.MarkOnboardingDone()
   ```
   The shipped dev `groqApiKey` is ~240 chars and does not start with `gsk_`, so it hits the **`>= 50` branch ‚Üí `hasKey = true` ‚Üí `MarkOnboardingDone()`** ‚Äî even though the user never entered a key. But that blob is DPAPI-encrypted to the **developer's** user+machine: on the new user's box, `DPAPIDecrypt` ‚Üí `CryptUnprotectData` fails ‚Üí **`return ""`** (`lib/dpapi.ahk:113-114`). So the app considers setup "all set", suppresses the wizard (the marker exists), and then **transcription fails on first real use with an empty/garbage key ‚Äî and the wizard never reappears.** This directly defeats the safety net the code documents at `onboarding_ui.ahk:171-172` ("Only mark onboarding complete if user has configured a valid API key; otherwise the wizard reappears"). Hard Dad-Test failure for 100% of fresh installs.
2. **`launchAtStartup=1` is inherited.** `onboarding_ui.ahk:340-341` only defaults it to `0` `if !cfg.Has("launchAtStartup")` ‚Äî the shipped config *has* the key (=1), so onboarding leaves it on. Every fresh install silently registers the `HKCU\‚Ä¶\Run\QuickSay` autorun (T1.3-025) without asking.
3. **Credential-blob hygiene + personal prefs as defaults.** Shipping any `groqApiKey` blob to all users is poor hygiene (even though it's cross-machine-undecryptable, so not a usable-key leak), and the dev's `widgetX/Y`, `historyRetention=500`, `soundTheme="default"` (T1.3-002), `lastSeenVersion`/`lastWeeklySummary` become the fresh-install baseline.

**Compounding:** `config.example.json` (CLAUDE.md's "all keys with defaults", the natural clean-default source) is **deleted from the working tree** (`Test-Path` ‚Üí False; `git status` shows `D config.example.json`).
**Fix:** Ship a pristine default ‚Äî generate it at build time (no `groqApiKey`, `launchAtStartup=0`, neutral widget pos, a real `soundTheme`) or restore `config.example.json` and `Source:` *that* to `config.json`. The absence of `groqApiKey` makes `OnClose()` correctly leave the wizard armed until the user enters a real `gsk_` key.
**Owner:** `owned by M.1` (build hygiene) ¬∑ cross-ref T1.6 (version fields) ¬∑ T1.4 (`soundTheme`).

> **`onlyifdoesntexist` flag itself is correct (no issue).** For the **upgrade** path the flag is the right choice: an existing user's `config.json` and `dictionary.json` are *not* overwritten (Inno skips files that already exist), so user settings/dictionary survive an in-place upgrade. The defect above is strictly about *what file* is shipped as the fresh-install seed, not about the flag.

### T1.3-003 ‚Äî `lib\*` ships uncompiled `.ahk` source + WebView2Loader.dll is triplicated ‚Äî **P3**

### T1.3-009 ‚Äî Patch releases collide on the installer filename (`vX.Y` only) ‚Äî **P2**

## Category B ‚Äî `release.ps1` idempotency & failure modes

### T1.3-010 ‚Äî Default auto-bump double-bumps on re-run ‚Äî **P2**
**File:** `release.ps1:39-62`.
**Evidence:** With no `-Version`, `Get-CurrentVersion` reads the *current* `localVersion` and adds 1 (`patch` default). Re-running after a successful run reads the **already-bumped** value and bumps again (1.9.0‚Üí1.9.1‚Üí1.9.2‚Ä¶). Only `-Version X.Y.Z` is re-run-stable (the regexes replace `\d+\.\d+\.\d+` with the same target).
**Fix:** Once `Development/VERSION` is the source of truth (T1.6), detect "already at target" and no-op; or require explicit `-Version` for re-runs.
**Owner:** `owned by T1.6`.

### T1.3-027 ‚Äî WebView2 bootstrapper needs internet; `ResultCode` unchecked ‚Üí silent failure on offline clean VM ‚Äî **P2**
**File:** `redist/MicrosoftEdgeWebview2Setup.exe` (**1.6 MB**, signed `CN=Microsoft Corporation`) + `setup.iss:195`.
**Evidence:** At 1.6 MB this is the **Evergreen *Bootstrapper*** (the standalone offline runtime installer is ~150 MB and named `MicrosoftEdgeWebView2RuntimeInstallerX64.exe`). The bootstrapper **downloads** the runtime from Microsoft at install time ‚Üí it **requires internet**. The `Exec(‚Ä¶ '/silent /install', ‚Ä¶, ResultCode)` call (`:195`) **ignores `ResultCode`**, so on a machine with no/blocked internet the runtime install fails silently, setup reports success, and the app then launches into a **blank/broken WebView2** (settings + onboarding are WebView2 UIs). This is exactly the clean-VM/air-gapped scenario M.2 must cover.
**Fix:** Check `ResultCode` and warn/abort with a clear message; for true offline support, bundle the standalone Evergreen runtime (size cost) or detect no-network and instruct the user. At minimum, surface failure instead of swallowing it.
**Owner:** `owned by M.1` / M.3 (offline-support decision).

### T1.3-028 ‚Äî Uninstall "keep data?" prompt: silent default keeps; "No" path `DelTree`s without second confirm ‚Äî **P3**
