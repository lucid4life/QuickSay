## Category A — Installer file integrity

### T1.3-001 — Installer ships the developer's live `config.json` (encrypted key + personal prefs) — **P1**
**File:** `setup.iss:72` (`Source: "config.json"; … Flags: onlyifdoesntexist`) → ships `Development/config.json`.
**Evidence:** `config.json` on disk is the dev's working config:
```json
"groqApiKey": "AQAAANCMnd8BFdERjHoAwE/Cl+sBAAAA…",   // DPAPI blob
"launchAtStartup": 1, "widgetX": 405, "widgetY": 705,
"lastWeeklySummary": "20260513002031", "lastUpdateCheck": "2026-05-19",
"historyRetention": 500, "lastSeenVersion": "1.9.0-beta", "soundTheme": "default"
```
On a **fresh install** (`onlyifdoesntexist` ⇒ dropped because the user has none yet) the new user inherits all of these. Two concrete harms:
1. **`launchAtStartup=1` is inherited.** `onboarding_ui.ahk:340-341` only defaults it to `0` `if !cfg.Has("launchAtStartup")` — but the shipped config *has* the key (=1), so onboarding leaves it on. Every fresh install therefore silently registers the `HKCU\…\Run\QuickSay` autorun (see T1.3-025) without asking.
2. **`groqApiKey` is a DPAPI blob bound to the dev's user+machine.** It cannot be decrypted on any other machine (DPAPI `Unprotect` fails), so it is *not* a usable-key leak — but the app sees a *present-but-invalid* key, which can short-circuit the "enter your free key" first-run path into a confusing "invalid API key" state (Dad-Test regression). Shipping any credential blob to all users is also poor hygiene.
**Compounding:** `config.example.json` (CLAUDE.md's "all keys with defaults", the natural clean-default source) is **deleted from the working tree** (`Test-Path` → False; `git status` shows `D config.example.json`).
**Fix:** Ship a pristine default — generate it at build time (no `groqApiKey`, `launchAtStartup=0`, neutral widget pos, `soundTheme` a real theme) or restore `config.example.json` and `Source:` *that* to `config.json`. Add the default-config emit to the build.
**Owner:** `owned by M.1` (build hygiene) · cross-ref T1.6 (version fields) · T1.4 (`soundTheme`).

### T1.3-002 — Shipped `soundTheme:"default"` matches no theme directory — **P3**
**File:** `config.json:24` (`"soundTheme": "default"`).
**Evidence:** `sounds/` contains only `bloom, click, crystal, mechanical, neon, subtle` (P0.2 confirmed 6 themes). There is **no `default` dir**. A fresh install inheriting this value (via T1.3-001) has a sound theme that resolves to nothing.
**Fix:** Default config should name a real theme (e.g. `subtle`). Verify the engine's fallback when `soundTheme` is unknown.
**Owner:** `owned by T1.4` (sound system) · ships via T1.3-001 fix.

### T1.3-003 — `lib\*` ships uncompiled `.ahk` source + WebView2Loader.dll is triplicated — **P3**
**File:** `setup.iss:82` (`lib\*` recursesubdirs).
**Evidence:** Installs readable source: `ComVar.ahk, dpapi.ahk, GDI.ahk, http.ahk, JSON.ahk, Promise.ahk, settings-ui.ahk, web-overlay.ahk, WebView2.ahk`. These are bundled into `QuickSay.exe` at compile time, so they are only needed for the `AutoHotkey64.exe`-based "fallback script" path (which also requires `onboarding_ui.ahk`, #5). `WebView2Loader.dll` ships **three times**: root `64bit\` (#11), `lib\` and `lib\64bit\` (#10).
**Severity rationale:** No secrets in `lib/` (the only credential is the DPAPI blob in `config.json`, T1.3-001). Source disclosure of an AHK app is low-impact (the compiled exe is trivially decompilable anyway). Pure footprint/clarity.
**Fix:** If the script-fallback path is dead, drop `onboarding_ui.ahk` + the `lib\*.ahk` sources and ship only the runtime DLL. De-duplicate WebView2Loader.dll.
**Owner:** `owned by T1.3-followup` / M.1.

**Files:** `data/changelog.json:3` (`"version":"1.9.0-beta"`), `config.json:14` (`"lastSeenVersion":"1.9.0-beta"`), vs `setup.iss:5` (`MyAppVersion "1.9.0"`) + `QuickSay.ahk:3398` (`localVersion := "1.9.0"`) + `QuickSay.ahk:3-5` (`SetFileVersion 1.9.0.0`, `SetProductVersion 1.9.0.0`, `SetProductName … v1.9`).
**Evidence — every version location and its current form:**
| Form | Locations |
|---|---|
| `1.9.0` | `QuickSay.ahk` `localVersion` (:3398); `setup.iss` `MyAppVersion` (:5) → feeds `AppVersion`/`AppVerName`/`VersionInfoVersion`/`VersionInfoProductVersion` |

### T1.3-009 — Patch releases collide on the installer filename (`vX.Y` only) — **P2**
**Files:** `setup.iss:24` (`OutputBaseFilename=QuickSay_Beta_v1.9_Setup`) + `release.ps1:517` (`$r2InstallerName = "QuickSay_Beta_v${shortVer}_Setup.exe"`).
**Evidence:** Both use **`$shortVer` (`X.Y`)**, not `$semVer`. A `1.9.0` and a `1.9.1` patch both emit `QuickSay_Beta_v1.9_Setup.exe`. STEP 5b `r2 object put` then **overwrites** the prior object at the same key, and the GitHub asset name is ambiguous across patches. `release.ps1` even hedges with fallback naming (`:478`, `:521`) because `$displayVer` becomes `v1.9.1` for patches while the ISCC output stays `v1.9`. Users who downloaded `v1.9` silently get a different binary after a patch under the same URL/hash expectation.
**Fix:** Use `$semVer` in `OutputBaseFilename` (via the version macro) and the R2 key for any patch>0, or always. Keep one stable filename scheme.
**Owner:** `owned by T1.6`.

## Category B — `release.ps1` idempotency & failure modes

### T1.3-010 — Default auto-bump double-bumps on re-run — **P2**
**File:** `release.ps1:39-62`.
**Evidence:** With no `-Version`, `Get-CurrentVersion` reads the *current* `localVersion` and adds 1 (`patch` default). Re-running after a successful run reads the **already-bumped** value and bumps again (1.9.0→1.9.1→1.9.2…). Only `-Version X.Y.Z` is re-run-stable (the regexes replace `\d+\.\d+\.\d+` with the same target).
**Fix:** Once `Development/VERSION` is the source of truth (T1.6), detect "already at target" and no-op; or require explicit `-Version` for re-runs.
**Owner:** `owned by T1.6`.

### T1.3-011 — Half-done run leaves uncommitted, bumped source with no rollback — **P1**
**File:** `release.ps1` STEP 1 (157-384) vs STEP 7b (794-900).
**Evidence:** STEP 1 mutates ~7 source files (`QuickSay.ahk`, `onboarding_ui.ahk`, `settings_ui.ahk`, `lib/settings-ui.ahk`, `gui/settings.html`, `setup.iss`, `LICENSE_AGREEMENT.rtf`) + `changelog.json` **before** compile/sign/upload. If the run dies at compile (`:411`), sign (the documented Azure hang, T1.3-016), ISCC (`:470`), or R2 (`:531`), those edits are already on disk. STEP 7b commits **only website files** — the **app-source version bumps are never committed by the script**. There is no `try/finally`, no `git stash` checkpoint, no rollback. Result: a Development tree dirtied with a bumped-but-unreleased version, possibly a half-built/unsigned `QuickSay.exe`, and a `changelog.json` entry for a version that never shipped. Cleanup is manual and easy to get wrong (cf. the parallel-session commit accident recorded in the T1.1 memory).
**Fix:** Snapshot before STEP 1 (`git stash` or record HEAD); wrap STEPs 1-8 so a failure restores source; or move version mutation *after* a successful compile+sign dry pass. At minimum, commit the app-source bumps on success so the tree is never left dirty-but-released.
**Owner:** `owned by T1.6`.

### T1.3-012 — `-Changelog` omission silently degrades, contradicting the mandate — **P2**
**File:** `release.ps1:358` (`if ($Changelog -ne "")`), `:566-570`, `:936-938`.
**Evidence:** CLAUDE.md: "**Always** pass `-Changelog`." But if omitted, the script proceeds with **no error and no prompt**: `changelog.json` is **not updated at all** (the whole block is skipped), and `version.json`/release notes fall back to the generic `@("Bug fixes and improvements")`. Nothing surfaces the omission.
**Fix:** Guard at the top — if `-Changelog` is empty and not `-DryRun`, prompt to confirm "ship with generic changelog?" (default No) or hard-require it for non-patch bumps.
**Owner:** `owned by T1.6`.

### T1.3-013 — GitHub release re-run deletes & recreates (loses metadata) — **P2**
**File:** `release.ps1:958-963`.
**Evidence:** `if (gh release view $tag) { gh release delete $tag --yes; sleep 2 }` then recreate. A re-run **destroys** the existing release's download counts, publish timestamp, and any manual edits/discussion. Combined with T1.3-009, a patch with the same `vX.Y` asset name worsens the confusion.
**Fix:** Prefer `gh release upload --clobber` on an existing release, or refuse to recreate without an explicit `-Force`.
**Owner:** `owned by T1.6`.

### T1.3-014 — External-dependency failure messages are uneven and machine-specific — **P2**
**File:** `release.ps1:82-89, 400-403, 459-462, 434-438, 443, 525, 912-915`.
**Evidence:** Ahk2Exe/ISCC missing → clear `Write-Fail` + `exit 1` (good). But `signtool`/`gh`/`npx`/`az` are assumed on PATH — a missing `signtool` throws a raw PowerShell error mid-sign; the Dlib path is hardcoded to **`C:\Users\abeek\…`** (breaks on any other machine/account). `az account show` (`:434`) only verifies a login *exists*, not that a `codesigning.azure.net` token is valid (see T1.3-016).
**Fix:** Preflight all required tools/paths with actionable messages; parameterize the Dlib path (env var or `signing/metadata.json`). Fold into the `--check-sync`/preflight gate.
**Owner:** `owned by T1.6`.

### T1.3-024 — Runtime-created files are orphaned on a normal uninstall — **P2**
**File:** `setup.iss` `[UninstallDelete]` (120-132) + uninstall `[Code]` (208-221).
**Evidence:** Inno's uninstall log only removes files it *installed* via `[Files]`. Files created at **runtime** — `data\history.json`, `data\statistics.json`, `data\quicksay.db`, `data\audio\QS_*.wav`, `data\logs\debug.txt` — are **not** in `[Files]` and **not** in `[UninstallDelete]`, so a standard uninstall (where the user answers **"Yes, keep settings"**, also the silent default) **leaves them behind** in `{app}\data`, and the non-empty `{app}\data` (and `{app}` itself) is therefore not removed → orphaned directory tree. **Fix:** Decide the policy explicitly. If data should be removable, add the runtime files/dirs to `[UninstallDelete]` (gated by the keep-prompt). If data should persist across reinstall, move it out of `{app}` (ties to T1.3-023).
**Owner:** `owned by M.1` (with T2.1 location decision).

### T1.3-025 — Orphaned `HKCU\…\Run\QuickSay` autorun after uninstall — **P1**
**File:** `lib/settings-ui.ahk:994-1018`:
```ahk
regKey := "HKCU\Software\Microsoft\Windows\CurrentVersion\Run"
regName := "QuickSay"
launcherPath := A_ScriptDir "\QuickSay.exe"
…
RegWrite('"' . launcherPath . '"', "REG_SZ", regKey, regName)   ; when launchAtStartup on
…
try RegDelete(regKey, regName)                                   ; when off
```
**Evidence:** The **app** (settings process) owns autorun via an `HKCU\…\Run\QuickSay` value → `"{app}\QuickSay.exe"`. The **installer** owns a *different*, independent mechanism — a `{userstartup}` Startup-folder shortcut (`setup.iss:112`, `Tasks: startupicon`, unchecked by default) which *is* removed by uninstall. The installer has **no knowledge of the `Run` value**, so uninstall does **not** delete it. **Fix:** Add `[UninstallRun]` to delete it — e.g. `Filename: "{cmd}"; Parameters: "/c reg delete ""HKCU\Software\Microsoft\Windows\CurrentVersion\Run"" /v QuickSay /f"; Flags: runhidden; RunOnceId: "DelRunKey"` — and/or have the app delete its own `Run` value on a clean exit / pre-uninstall hook. (And fix T1.3-001 so fresh installs default `launchAtStartup=0`.)
**Owner:** `owned by M.1`.
